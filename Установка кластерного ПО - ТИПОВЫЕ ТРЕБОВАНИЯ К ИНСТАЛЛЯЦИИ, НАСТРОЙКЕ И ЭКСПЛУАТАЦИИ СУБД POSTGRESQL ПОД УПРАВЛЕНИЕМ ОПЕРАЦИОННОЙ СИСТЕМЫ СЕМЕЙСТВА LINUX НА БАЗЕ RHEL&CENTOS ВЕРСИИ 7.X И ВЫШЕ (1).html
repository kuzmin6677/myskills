    Кластерная конфигурация БД¶

    Установка и настройка кластерного ПО¶

            1. Выполнить установку кластерного ПО на всех узлах кластера: 

    sudo yum install pcs corosync pacemaker resource-agents

            2. Выполнить смену пароля учетной записи hacluster на всех узлах кластера: 

    sudo passwd hacluster

    Подсказка

    Пользователь hacluster создается в операционной системе автоматически

            3. Заменить на всех узлах кластера БД файл pgsql размещенный в каталоге /usr/lib/ocf/resource.d/heartbeat/ на аналогичный выложенный по адресу http://link.rzd/6885

    Внимание

    "Проверить, что файл имеет права -rwxr-xr-x root root"

            4. Выполнить запуск кластера и активацию служб кластерного ПО на всех серверах: 

    sudo systemctl start pcsd.service; 
sudo systemctl is-active pcsd.service; 
sudo systemctl enable pcsd.service; 
sudo systemctl is-enabled pcsd.service

            5. Выполнить первоначальную авторизацию узлов кластера с любого из серверов:

    sudo pcs cluster auth node1 node2 

    sudo pcs host auth -u hacluster node1 node2

    node1 node2 - DNS-имена серверов или указанные в файле /etc/hosts

            6. Выполнить первоначальную конфигурацию кластера СУБД с любого узла:

    sudo pcs cluster setup --start --name cluster_name node1 node2

    sudo pcs cluster setup cluster_name  node1 node2

    Внимание!

    Подпункты 6.1 и 6.2 выполняются только для pacemaker 2.0

            6.1. Выполнить очистку cib на всех узлах кластера 

    sudo rm -rf /var/lib/pacemaker/cib/*

            6.2. Выполнить запуск кластера pcs на всех узлах 

    sudo pcs cluster start

            7. Добавить параметр token в файле /etc/corosync/corosync.conf в тело totem на всех узлах кластера, ДРУГИХ ИЗМЕНЕНИЙ В БЛОК totem ВНОСИТЬ НЕ ТРЕБУЕТСЯ: 

    totem {
version: 2    
cluster_name: cluster     
secauth: off     
transport: udpu     
token: 4000    
}

    Дополнительная настройка отказоустойчивости при инсталляции кластера БД на виртуальные машины среды VMware¶

            Ввести дополнительные настройки в конфигурацию кластера выполнив команды ниже с любого узла: 

    sudo pcs property set stonith-enabled="true" &&

sudo pcs stonith create fence-agent-pg2 fence_vmware_soap \
ipaddr=gvc-vcenter-01.vmware.rzd ipport=443 ssl_insecure=1 inet4_only=1 \
login="gvcxxx@vsphere.gvc" passwd="xxxxxxx" \
pcmk_reboot_action=reboot \
pcmk_host_map="name1:gvc-xxx-xxx" \
pcmk_host_check=static-list \
pcmk_host_list="gvc-xxx-xxx-1" \
power_wait=3 op monitor interval=60s timeout="180s" on-fail="ignore" --force &&

sudo pcs stonith create fence-agent-pg1 fence_vmware_soap \
ipaddr=gvc-vcenter-01.vmware.rzd ipport=443 ssl_insecure=1 inet4_only=1 \
login="gvcxxx@vsphere.gvc" passwd="xxxxxxx" \
pcmk_reboot_action=reboot \
pcmk_host_map="name2:gvc-xxx-xxx" \
pcmk_host_check=static-list \
pcmk_host_list=" Vmware_host_name1,Vmware_host_name2" \
power_wait=3 op monitor interval=60s timeout="180s" on-fail="ignore" --force &&

sudo pcs constraint location fence-agent-pg2 avoids pg1=INFINITY &&
sudo pcs constraint location fence-agent-pg1 avoids pg2=INFINITY

    fence-agent-pg1 и fence-agent-pg2 – имя ресурса fence-agent на узлах кластера БД;
    ipaddr=gvc-vcenter-01.vmware.rzd – доменное имя консоли VMware;
    login="gvcxxx@vsphere.gvc" passwd="xxxxxxx" – логин и пароль учетной записи в консоли VMware (необходимо запросить у отдела, ответственного за сопровождение платформы виртуализации VMware);
    pcmk_host_map="name1:gvc-xxx-xxx" и pcmk_host_map="name2:gvc-xxx-xxx" – сопоставление серверов БД node1 и node2, указанных при настройке кластера в п. Настройки ресурсов кластера, и имен виртуальных машин в VMware;
    pcmk_host_list="Vmware_host_name1,Vmware_host_name2" – имена виртуальных машин в VMware.

    Масштабирование кластера БД¶

    Удаление узла из кластера БД

            1. Выполнить удаление сервера из кластера командой: 

    sudo pcs cluster node remove <name_node>

            2. Инициировать перечтение конфигурации кластера corosync: 

    sudo pcs cluster reload corosync

            3. Выполнить корректировка параметров ресурса pgsql в кластере: 

    sudo pcs resource update pgsql node_list="pg1, pgN"
sudo pcs resource cleanup

    node_list="pg1, pgN" – перечисляем только те узлы кластера, которые остаются.

    Добавление узла в кластер

            1. Перед выполнением процедуры добавления нового узла в кластер БД, на новом сервере необходимо выполнить все подготовительные операции описанные в разделелах:
            Минимальные требования к операционной системе LINUX
            Настройка локального брандмауэра iptables для кластерной конфигурации БД

            2. После подготовки сервера к включению его в кластер, необходимо выполнить авторизацию кластерного ПО на всех узлах кластера БД включая новый путем выполнения на master-узле команды: 

    sudo pcs cluster auth pg1 pg2 ... pgN

            3. Инициировать перечтение конфигурации кластера corosync: 

    sudo pcs cluster reload corosync

            4. Выполнить корректировка параметров ресурса pgsql в кластере: 

    sudo pcs resource update pgsql node_list="pg1 pg2 new_node_name "
pcs cluster node add new_node_name

            5. Выполнить запуск кластерного ПО на добавленном узле кластера: 

    sudo pcs cluster start

    Замена старого узла в кластере на новый

            1. Перед выполнением процедуры добавления нового узла в кластер БД, на новом сервере необходимо выполнить все подготовительные операции описанные в разделелах:
            Минимальные требования к операционной системе LINUX
            Настройка локального брандмауэра iptables для кластерной конфигурации БД

            2. После подготовки сервера к подмене им текущего узла в кластере, необходимо выполнить удаление сервера, который требуется заменить командой: 

    sudo pcs cluster node remove <name_node>

            4. Выполнить корректировку файла /etc/hosts на всех узлах кластера заменив удаленный сервер на новый

            5. Выполнить авторизацию кластерного ПО на всех узлах кластера БД, включая новый, путем выполнения на master-узле команды: 

    sudo pcs cluster auth pg1 pg2 new_node_name

            6. Инициировать перечтение конфигурации кластера corosync: 

    sudo pcs cluster reload corosync

            7. Выполнить копирование БД с master-узла на новый: 

    pg_basebackup -h pg1 -U replica -D /pgdata/data -X s -P

            8. Выполнить корректировка параметров ресурса pgsql в кластере: 

    sudo pcs resource update pgsql node_list="pg1 pg2 new_node_name"
pcs cluster node add new_node_name

            9. Выполнить перезапуск кластерного ПО на добавленном узле кластера: 

    sudo pcs cluster stop
sudo pcs cluster start

    Управления кластером серверов базы данных СУБД PostgreSQL¶

            Запустить кластер на сервере: 

    sudo pcs cluster start

            Остановить кластер на сервере: 

    sudo pcs cluster stop

    Для полной остановки кластера, на последнем запущенном узле при выполнении данной команды необходимо дописать ключ --force

            Посмотреть параметры ресурса кластера БД: 

    sudo pcs resource show resource_id

            Обновить или дополнить параметры ресурса кластера БД: 

    sudo pcs resource update resource_id op start start-delay="3s" interval=0s timeout=90

    При использовании команды update необходимо указывать весь список параметров для обновляемого ресурса, включая существующие, так как данная команда выполняет перезапись всех параметров, и в случае неявного указания ряда параметров, их значения будут перезаписаны на значения «по умолчанию».

            Перезапустить ресурс кластера БД: 

    sudo pcs resource restart resource_id node#

    resource_id - имя ресурса базы данных
    node# - имя узла кластера БД, на котором необходимо выполнить перезапуск ресурса, например pg1

    Внимание!

    При перезапуске clone-ресурса кластера БД необходимо явно указывать узел, иначе перезапуск ресурса будет выполнен во всем кластере.

            Посмотреть и сбросить счётчик ошибок для конкретного ресурса кластера БД: 

    sudo pcs resource failcount show resource_id node#
sudo pcs resource failcount reset resource_id node#

    Эти команды необходимо выполнять в случае, если ресурс кластера находится в состояние ошибки и Pacemaker ждёт действий администратора, чтобы продолжить работу с ресурсом.

            Сбросить ошибки и статус ресурса кластера БД: 

    sudo pcs resource cleanup resource_id

    После выполнения данной команды Pacemaker продолжает работу с ресурсом.

            Активировать/деактивировать ресурс кластера БД: 

    sudo pcs resource disable resource_id 
sudo pcs resource enable resource_id

    Выполнение данных команд приводит к остановке/запуску соответствующей службы или процесса, в том числе fence-агента или СУБД PostgreSQL.

            Посмотреть полный вывод ограничений и зависимостей существующих в кластере БД на данный момент: 

    sudo pcs constraint --full

    В случае сбойных ситуаций и некорректной работы кластерного ПО с ресурсами кластера, например, некорректное размещение master-роли и slave-роли по узлам, выполнение данной команды позволяет свериться с исходной конфигурацией кластера БД.

            Выполнить удаление ограничения и зависимости кластера БД: 

    sudo pcs constraint remove constraint_id

            Сохранить текущую конфигурацию кластера БД: 

    sudo pcs cluster cib > /tmp/cluster.xml

    Применяется для резервного копирования конфигурации кластера.

            Применить новую конфигурацию кластера БД: 

    sudo pcs cluster cib-push /tmp/cluster.xml source lang="bash"

    Применяется для восстановления конфигурации кластера из резервной копии.

            Жесткая привязка ресурса кластера БД на конкретном узле: 

    sudo pcs resource ban ms-pgsql node#

    ms-pgsql – имя кластерного ресурса
    node# - имя узла кластера, на котором необходимо выполнить привязку, например pg2

            Просмотр статуса кластера БД в режиме реального времени: 

    sudo crm_mon -Art
