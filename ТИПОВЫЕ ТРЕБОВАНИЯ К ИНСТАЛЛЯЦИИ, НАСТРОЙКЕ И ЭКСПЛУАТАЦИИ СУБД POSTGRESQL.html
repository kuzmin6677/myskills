    Подготовка сервера к инсталляции СУБД PostgreSQL¶

            В качестве операционной системы необходимо использовать CentOS 7.x, в соответствии с распоряжением №ГВЦ-147/р от 14.05.2019г. Актуальная версия документа "Типовые требования к настройкам серверных ОС CentOS" размещена на портале ГВЦ в разделе ЦК ПТК и доступна по ссылке http://link.rzd/6826.

    Правила именований серверов базы данных¶

            Для именования серверов СУБД PostgreSQL применяется следующие правила:

            Для одноузловой конфигурации: 

    <Domain name>-<Functions Server>-DB-<Node number>

            Для кластерной конфигурации из двух узлов БД: 

    <Domain name>-<Function server>-CL-DB-<Node number>  
<Domain name>-<Function server>-CL-DB-<Node number>

    • DNS - имя домена (3-4 символа) Active Directory, в который включен сервер, например: gvc, orw;
    • Function server - указывается краткое именование АС согласно АСУ Реестр. При необходимости дополняется суффиксом с именем подсистемы АС, к которой данный сервер относится. (3-7 символов);
    • Node number - порядковый номер сервера. В рамках настроенной репликации данных master-сервер всегда обозначается номером 01.

    Обязательная настройка сервера к инсталляции СУБД PostgreSQL¶

            Каталоги хранения базы данных и ее резервных копий должны располагаться в отдельных созданных каталогах на независимых физических дисках. Директория для хранения БД должна иметь объем достаточный для размещения корневого каталога БД с перспективой роста не менее 2 лет.

    Внимание!

    Оценка перспективы роста БД по годам выполняется разработчиком.

            Для обеспечения работы автоинсталлятора СУБД перед подготовкой файловых систем сервера необходимо создать системную учетную запись на операционной системе сервера:

    Внимание!

    В случае работы под системными учетными записями «adminptk, adminvp0» выполнять пункты 1,2 не требуется.

            1. Создать пользователя: 

    sudo useradd <new_user>

            2. Назначить созданному пользователю пароль: 

    sudo passwd <new_user>

            3. Создать для нового пользователя файл sudoers с описанием прав работы на сервере: 

    sudo touch /etc/sudoers.d/users_no_pass

            4. Добавить строку в файл, созданный в предыдущем пункте: 

    echo "<new_user> ALL=(ALL) NOPASSWD: ALL" >> /etc/sudoers.d/users_no_pass

    Внимание!

    База данных и её резервные копии должны располагаться на отдельных смонтированных файловых системах

            Общие требования к файловым системам сервера:

    Имя Назначение Формат
    vg_pgdata Имя VG для хранения БД
    lvpgdata Имя LV для хранения БД
    /pgdata/ Имя тома хранения БД ext4
    vg_pgbackup Имя VG для хранения резервных копий БД
    lvpgbackup Имя LV для хранения резервных копий БД
    /backups_pg/ Имя тома хранения резервных копий БД ext4

            Организация каталогов базы данных на сервере:

    Имя каталога Описание
    /pgdata/data/ Хранение данных БД
    /backups_pg/ Хранение резервных копий БД
    /backups_pg/archive/ Хранение архивных резервных копий БД
    /backups_pg/archive_xlog/ Хранение архивов с транзакционными логами БД
    /var/log/pg_log/ Хранение лог-журналов работы СУБД
    /usr/pgsql-xx/ Хранение исполняемых файлов, библиотек и расширений, необходимых для работы СУБД PostgreSQL
    Имя каталога Описание
    /pgdata/data/ Хранение данных БД
    /backups_pg/ Хранение резервных копий БД
    /backups_pg/pg_probackup/backups Хранение архивных резервных копий БД
    /backups_pg/pg_probackup/wal Хранение архивов с транзакционными логами БД
    /var/log/pg_log/ Хранение лог-журналов работы СУБД
    /opt/pgpro/ Хранение исполняемых файлов, библиотек и расширений, необходимых для работы СУБД Postgres Pro

            Детальная настройка файловых систем представлена в разделе "Настройка дисков под установку СУБД PostgreSQL".

            Для обеспечения возможности восстановления базы данных на точку времени (согласно ТЗ/ЧТЗ требование к значению допустимого RPO), при расчете объема базы данных необходимо учитывать суточный объем логов транзакций.

    Внимание!

    Данная информация должна быть предоставлена разработчиком системы.

    Объем, достаточный для хранения резервных копий (С) вычисляется по формуле: 

    С = А * Б

    А – объем директории для хранения БД
    Б – количество резервных копий базы данных необходимое для хранения исходя из степени критичности системы и требований по сохранности информации при авариях, описанных в ТЗ/ЧТЗ.

            В случае настройки кластерной конфигурации СУБД, рекомендуется в файле /etc/hosts указывать короткие имена pg1 и pg2 для узлов кластера: 

    #127.0.0.1 test    
192.168.0.2 RZD-TESTAS-CL-DB-01 pg1 
192.168.0.3 RZD-TESTAS-CL-DB-02 pg2

            На обоих узлах кластера необходимо проверить, что в качестве кодировки установлена UTF8: 

    cat /etc/locale.conf

    Настройка диска под корневой каталог СУБД PostgreSQL /pgdata¶

            1. Выполнить команду «fdisk –l», в выводе найти интересующий диск: 

    sudo fdisk -l

            2. Перед использованием диска или раздела в качестве физического тома необходимо его инициализировать: 

    sudo pvcreate /dev/sdb

            3. Создание новой Volume Group vg_data, и отдаем для использование раздел /dev/sdb: 

    sudo vgcreate vg_data /dev/sdb

            4. Открываем новый Logical Volume в vg_data и отдаем все свободное место в выделенной vg_data под данный раздел: 

    sudo lvcreate -l100%FREE -n lvpgdata vg_data

            5. Форматируем выделенное место в lvpgdata в формат ext4: 

    sudo mkfs.ext4 /dev/vg_data/lvpgdata

            6. Добавляем данный раздел в автомонтирование fstab к директории /pgdata: 

    sudo echo "/dev/vg_data/lvpgdata /pgdata ext4 defaults 0 0" >> /etc/fstab

            7. Выполнить перечитывание конфигурации fstab: 

    sudo mount -a

    Настройка дисков под создание резервных копий (Backup) /backups_pg¶

            1. Выполнить команду «fdisk –l», в выводе найти интересующий диск: 

    sudo fdisk -l

            2. Перед использованием диска или раздела в качестве физического тома необходимо его инициализировать: 

    sudo pvcreate /dev/sdc

            3. Создание новой Volume Group vg_data, и отдаем для использование раздел /dev/sdc: 

    sudo vgcreate vg_pgbackup /dev/sdc

            4. Открываем новый Logical Volume в vg_data и отдаем все свободное место в выделенной vg_data под данный раздел: 

    sudo lvcreate -l100%FREE -n lvpgbackup vg_pgbackup

            5. Форматируем выделенное место в lvpgdata в формат ext4: 

    sudo mkfs.ext4 /dev/vg_pgbackup/lvpgbackup

            6. Добавляем данный раздел в автомонтирование fstab к директории /pgdata/backups: 

    sudo echo "/dev/vg_pgbackup/lvpgbackup /backups_pg ext4 defaults 0 0" >> /etc/fstab

            7. Выполнить перечитывание конфигурации fstab: 

    sudo mount -a

    Настройка ядра операционной системы¶

    Внимание!

    Если не выполнен данный пункт, то велика вероятность падение экземпляра СУБД из-за остановки процесса операцией OMM Killer

    Дальнейшие действия выполняются с привилегиями пользователя root

            Добавить в настройки ядра /etc/sysctl.conf строки по ограничению использования памяти: 

    vm.overcommit_ratio = 100
vm.overcommit_memory = 2

            Выполнить перечитывание конфигурационного файла: 

    sysctl -p

    Внимание!

    В операционной системе сервера в конфигурационном файле sysctl.conf может быть определен ряд настроек и их значений по умолчанию. Для корректной работы СУБД рекомендуется закомментировать данные настройки. Любые параметры и их значения, не указанные в данном документе и не предустановленные по умолчанию, должны быть согласованы с разработчиком системы.
